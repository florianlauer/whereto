<task type="auto">
  <name>Task 1: Initialize Supabase local dev and create database schema migration</name>
  <files>devenv.nix, supabase/config.toml, supabase/migrations/00001_initial_schema.sql, supabase/seed.sql</files>
  <action>
1. Add `supabase-cli` to `devenv.nix` packages list (per user decision: installed via devenv.nix, not npm).

2. Run `supabase init` in the project root to create `supabase/` directory with `config.toml`. If supabase dir already exists, skip.

3. Create `supabase/migrations/00001_initial_schema.sql` with the full schema from the RESEARCH.md Pattern 4 section. This includes:
   - `profiles` table: id (uuid PK, FK to auth.users ON DELETE CASCADE), created_at, updated_at
   - `wishlists` table: id (uuid PK, gen_random_uuid), user_id (FK to profiles ON DELETE CASCADE), created_at, updated_at, UNIQUE(user_id)
   - `wishlist_items` table: id (uuid PK, gen_random_uuid), wishlist_id (FK to wishlists ON DELETE CASCADE), poi_id (text NOT NULL), country_code (text NOT NULL), position (integer NOT NULL DEFAULT 0), created_at, updated_at, UNIQUE(wishlist_id, poi_id)
   - RLS enabled on all three tables
   - RLS policies: users can only SELECT/UPDATE own profile; users can only SELECT/UPDATE own wishlists; users can SELECT/INSERT/UPDATE/DELETE own wishlist_items (ownership checked via wishlist -> user_id chain)
   - Use `(select auth.uid())` wrapper (not bare `auth.uid()`) in all policies for performance
   - INSERT policies use WITH CHECK, not USING
   - `handle_new_user()` trigger function: SECURITY DEFINER, inserts into profiles then wishlists on auth.users INSERT
   - `handle_updated_at()` trigger function: sets new.updated_at = now(), applied to all three tables

4. Create `supabase/seed.sql` with test data:
   - 2-3 test users created via `auth.users` (use Supabase's auth schema pattern for local seed)
   - Note: The trigger will auto-create profiles and wishlists for seeded users
   - Add sample wishlist_items for each test user (3-4 items with real poi_ids from the existing pois.json data if possible, otherwise use placeholder ids like "poi_paris_eiffel")

5. Start Supabase local: `supabase start` (requires Docker running)
6. Apply migration: `supabase db reset` (applies migrations + seed)
7. Verify tables exist: `supabase db lint` should pass with no errors
   </action>

   </verify>
   <done>All three tables created with RLS enabled, trigger fires on auth.users INSERT creating profile + wishlist rows, seed data loaded successfully, `supabase db reset` completes without errors.</done>
   </task>
